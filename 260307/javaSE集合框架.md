### 1. 普通数组
    1. java中只有值传递
        当数组作为函数参数被传递时，本质上传递的是数组的引用（内存所占空间的第一个地址），例如传递int[] mylist, java 传递的是引用的值（地址副本），因此可以修改数组内容，但不能改变引用指向。但mylist[0]可以指向对应的值，因为这是用地址+偏移的方式访问的，就是原来数组的值。
    2. hot100常用方法：
        list.length 获取数组长度
        list.sort() 排序

### 2. ArrayList
    ArrayList本质是升级版的普通数组，解决了普通数组无法动态扩容、增删复杂、无实用API的问题。
    1. 自动扩容
        ArryayList底层的堆空间仍然是一块连续的空间，get方法中使用baseAddress+index*size进行访问，所以get方法时间复杂度是O(1)
        如果扩容（默认1.5倍，newCapacity = oldCapacity + (oldCapacity >> 1)），则需要重新分配一块新的内存空间，并把原数组中的元素复制过去，时间复杂度是O(n)
    2. 使用API简化操作
        尾插，可能需要扩容。
        中间插入，需要找到插入位置，然后插入、后续元素批量后移。删除类似。
        尽管原理不变，但ArryayList增删改查都有了对应的API，极大简化了操作。
    
### 3. LinkedList
    LinkedList底层是双向链表，解决了连续内存的数组在插入删除效率极低的问题。
    1. 节点定义
        数据、前驱指针、后驱指针。
    2. 优点：更快的增删
        头尾的增删速度更快，但中间插入、删除速度更慢。（因为需要遍历查询到对应位置）
    3. 代价：更慢的查询
        判断头尾加速查找，但仍然存在时间复杂度O(n)。

### 4. HashMap
    1. 底层结构：数组+链表/红黑树（jdk1.8）
        1.第一层：数组
            数组的索引是索引值（键进行扰动计算后得出的值），对应的值是一个链表或者红黑树的头节点地址。
            数组长度为2的幂次方，index = hash & (length - 1)，相比于%运算更快更均匀，保证低位均匀分布。
        2.第二层：链表/红黑树
            链表：链表节点的next指针指向下一个节点，hash值相同的节点会放在同一个链表中。如果不使用红黑树，头节点一般不会变化，第一层数组存起来的地址也不会变（除非扩容导致的链表拆分）。
            红黑树：链表太长会导致查找效率下降，使用红黑树加速查找。红黑树的头节点可能变化，所以对应第一层数组的值也会变化。
    2. 哈希冲突
        1. 必然会出现的鸽巢原理、哈希函数输出范围有限且客观上哈希函数的确会出现重复输出，总之就是两个不一样的key会映射到同一个数组索引上
        2. 存储上，可以接入到链表的尾部（JDK8尾插，之前是头插），查找时遍历。但过于长的链表需要优化为红黑树（链表长度阈值为8，数组容量阈值为64否则优先扩容）
        3. 查找时，找到数组索引后进入第二层，遍历查询或是查询红黑树。
    3. 扩容（增加key）
        1. 如果不扩容，key少链表长，查找变慢
        2. 触发扩容的条件：阈值 = 默认容量*负载因子
        3. 扩容方法：
            1. JDK7经典实现：创建容量翻倍的新数组，旧数组转移，替换地址等，需要把每个节点重新哈希计算索引然后放入，可能会导致结构的超大变化
            2. JDK8改进：创建容量翻倍的新数组，旧数组转移，替换地址等，但是新数组的索引计算方式不同，2倍扩容导致二进制的多出第一位1和0可以实现区分。单个节点重新计算和jdk7一样，红黑树直接拆分为2簇，链表看首位（0留，1变）
            3. 负载因子0.75是符合泊松分布的经验取值。
    4. 红黑树
    链表的查询就是普通的遍历，很慢，但如果改为数组，维护十分艰难，因此引入了树形结构平衡了查询和维护。而其中，红黑树就是对平衡二叉树AVL树的进一步优化。
        1. AVL树严格平衡，旋转频繁，红黑树选择了近似平衡，插入删除更加高效
        2. 红黑树的实现：
            1. 旋转：左旋右旋，与AVL一样
            2. 插入：先按二叉查找树正常插入，然后修复红黑树。修复时，注意父节点颜色。
                1. 黑色父节点，无需调整
                2. 红色父节点，变色/旋转

### 5. ConcurrentHashMap
ConcurrentHashMap 是 Java 并发包（java.util.concurrent）中提供的线程安全的哈希表实现。它在保证线程安全的同时，支持高并发读写，是 HashMap 的线程安全版本，但性能远优于 Hashtable。
    1. 线程安全，支持高并发读写
        普通的HashMap线程不安全，可能会导致链表链接有错误出现死循环。
    2. 键值不允许存在null
        避免二义性：键是否真的存在？（存在但key为null，还是根本没有这个key？）
    3. 读操作不锁，写操作只锁桶
        有Hashtable这样的全表锁，一次只能有一个线程进行操作，稳定但很慢。
        HashMap只锁一小部分的写，粒度比较小，可以让互不干涉的多个线程同时进行操作提高效率。
        1. JDK1.7：大锁拆小锁
        2. JDK1.8：无锁化+细粒度锁，能用CAS（乐观锁，竞争处理，谁抢到就是谁的，因为低竞争各自都有位置）的地方就不锁，必须锁的地方只用synchronized锁头节点，用volatile保证可见性。
    4. 迭代器弱一致性，不报ConcurrentModificationException
        1. 迭代器：使用迭代器，我们可以逐个访问集合中的元素（单向），而不需要使用传统的 for 循环或索引。这种方式更加简洁和灵活，并且适用于各种类型的集合。
        2. 使用迭代器遍历集合时，如果在遍历过程中对集合进行了修改（例如添加或删除元素），可能会导致 ConcurrentModificationException 异常。单线程通常使用remove()方法来安全删除元素，但多线程里，迭代线程和删除线程不互通，而remove只能更正自己的数据。
        3. 弱一致性代表着，就算迭代线程中其他线程动了数据，迭代线程也会继续工作，尽管数据可能不对劲。
    5. JDK1.8多线程扩容：多个线程领养一个迁移任务，各自负责一段桶的重新分配。

### 6. 个人思考
    1. 解决linkedlist查询慢的问题：修改linkedlist的底层，加入一个hashmap记住每个节点的索引和对应地址，插入删除时修改前驱后驱，同时维护hashmap
        优点：加速了get、set操作
        缺点：indexOf没有变化，还是要遍历询问你是不是等于该值
        更多的缺点：内存占用大，插入删除需要维护大量后续节点的索引
        已有的替代：SkipList跳表、红黑树、LinkedHashMap（保留插入顺序）、缓存最近访问
    2. HashMap多维数组实现：使用普通数组/类似ArrayList动态扩展的二维数组，跳过第一层
        优点：查询较快
        缺点：维护极难。若分布不均匀，需要经常扩容，频繁插入会导致大量数据移动，都不如红黑树
        已有的替代：开放地址法（冲突严重时性能爆降，且删除复杂）
