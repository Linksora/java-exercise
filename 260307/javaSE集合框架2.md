关键词：HashSet、LinkedHashMap、TreeMap、Set去重原理、LRU缓存思想

### 1. HashSet
    数学上，一个无重复、快速判断是否包含的集合是需要的，也可以抽象许多现实的场景，因此有其存在的必要性。
    替代性上，List允许重复元素，需要手动contain检查，需要O(n)，而使用哈希可以实现O(1)的增删改查。
    1. 底层基于HashMap实现，存储不重复的元素，但不保证元素顺序。底层中的那个哈希图只有key是我们真正要用的，而key对应的值，为一个固定的占位对象
    2. 有HashMap的所有优点。key唯一性保证hashset不重复，查找极快，有动态扩容，而位置由哈希散列决定。
    3. 常用方法
        myhashset.add(E,e) 添加元素
        myhashset.remove(E,e) 删除元素
        myhashset.contains(E,e) 判断是否包含元素
        myhashset.clear() 清空集合
        myhashset.iterator() 获取迭代器

### 2. LinkedHashMap
    HashMap是无序的（迭代得到的元素顺序是随机的，与放入顺序无关），但插入顺序有时需要可预测，比如实现LRU最近最少使用缓存，所以结合链表和哈希表，产生了LinkedHashMap。
    1. 结构上，LinkedHashMap的哈希表部分是普通哈希表，扰动后找到索引，然后索引后面是单向链表/红黑树。然后，在另一个位置放一个双向链表，将哈希表中的各个node双向链接起来。
    2. 从前往后是插入顺序，从后往前是访问顺序。所以需要双向链表。
    3. 优点：插入顺序，访问顺序，查找极快，有动态扩容。
    4. 使用方法new LinkedHashMap<>(16,0.75f,true)，第三个参数false为按插入顺序，true为访问顺序

### 3. TreeMap
    1. TreeMap 是 Java 集合框架中的一个类，实现了 NavigableMap 接口，底层基于红黑树实现。它根据键的自然顺序或自定义比较器对键进行排序，保证所有键值对处于有序状态。
    2. 键自动排序（自定义/自然顺序），基于红黑树所以时间复杂度为O(logn)
    3. 不允许出现null的key
    4. 常用方法包括导航方法（即怎么排）
    5. LinkedHashMap只解决了插入/访问顺序的问题，但有时需要按大小排序，TreeMap就派上了用场。

### 4. Set去重原理
    1. HashSet底层实现是HashMap，所以去重原理是HashMap的去重原理。
        1. hashCode快速定位桶，但hashcode冲突时不一定key相同
        2. 桶中equals比较，hashcode相同的是否真的key相同
        3. hashcode（与已有的每一个都）不相同，则肯定不一样，新加入；hashcode冲突，key但不一定真的相同，所以才需要equals比较。
    2. 使用上，三种set的比较

| 特性          | HashSet  | LinkedHashSet    | TreeSet             |
| ------------- | -------- | ---------------- | ------------------- |
| 底层实现      | HashMap  | LinkedHashMap    | TreeMap（红黑树）   |
| 顺序          | 无序     | 插入顺序         | 自然顺序/比较器顺序 |
| 时间复杂度    | O(1)     | O(1)             | O(log n)            |
| 是否允许 null | 允许一个 | 允许一个         | ❌ 不允许（需比较） |
| 线程安全      | ❌       | ❌               | ❌                  |
| 适用场景      | 通用去重 | 需要保留插入顺序 | 需要排序            |

### 5. LRU缓存思想
    LRU（Least Recently Used，最近最少使用） 是一种缓存淘汰策略。它的核心思想是：如果数据最近被访问过，那么将来被访问的概率也更高。当缓存满时，优先淘汰最久未被访问的数据。
    1. LRU是自下而上的经验总结，效果＞理论，利用了程序的局部性原理。
    2. 一般用LinkedHashMap实现，因为LRU需要记录访问顺序，而LinkedHashMap就是哈希表+链表，做到了去重+顺序。

### 6. 个人思考
    1. 关于java包装类型的历史遗留问题：如果一开始就把所有类型设计成引用类型？
        基本类型（如int），不是对象，没有方法，不能用于泛型。
        引用类型（如integer），以上皆可。
        泛型的实现方式是类型擦除，尽管写的时候可能是Boolean、integer之类的，但底层都会变成Object。
        这件事本质的原因就是java是一个需要编译的语言，编译会对数据类型进行一些预处理，最开始没有考虑到方法会适配不同的数据类型，而且直接把不同数据类型编译死会非常节约空间，因此一开始的基本类型无法保留这个余地。后来，为了实现不同类型的适配，使用了Object，编译时编译器只认Object，而把具体的类型当作参数处理，就避免了底层编译时编译了int没法编译float（或者重复编译两次）的问题。
        那么，如果java从一开始就用引用类型解决一切，彻底拒绝基本类型会让这个语言变得简洁吗？
        优点：更简洁，在功能上适配目前绝大部分使用场景。
        缺点：占用空间巨大。基本类型和应用类型还有一个区别，就是存储方式不同。
            1. 基本类型是在栈内存中直接存入，而引用类型是在栈内存中有一个对象，其中存的是地址，地址对应的另一部分可能在堆的空间才是值。也就是说，引用类型会占用更多空间。
            2. 除了存在时有内存压力，对临时变量的回收时也很复杂。基本类型直接从栈上操作即可，而引用类型要走通一套创建对象-创建值-回收值-回收对象的流程，会频繁STW（JWT中系统暂停所有应用程序线程），性能减弱。
    2. 那么，把Object的实现归纳到所有类型是否可行？
        优点：避免了内存压力，取消了引用类型
        缺点：底层实现十分复杂，不如设计引用类型，引用类型还可以提供简单的拆装箱，使用时切换简单，实现了既要有要。
        转折：但，如果这可以实现，就解决了引用类型带来的对象麻烦。这是在底层实现一次前人栽树，后来开发后人乘凉的计划。
        再转折：我好像以为我很聪明，但java开发团队显然也想到了，他们正在做Project Valhalla……

    4. LRU思考
        在计算机组成原理、操作系统等课程中LRU是经常见到的内容，一般在“存储空间有限+频繁调用”的场景中出现，在一小块高速内存放入最高频使用的数据，就可以最大化提高效率。但是如果单纯的LRU可能会被少次且大块的数据占用，是否可以单独把这种内容标记作黑名单，不占用LRU？
        1. 2Q，两个队列区分第一次访问、多次访问
        2. LIRS两次访问间隔判断冷热
        3. ARC动态平衡
        4. Midpoint Insertion Strategy（MySQL中，新数据插入中间，更容易被淘汰）
