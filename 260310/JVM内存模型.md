### 1. JVM运行时的内存结构
    JVM内存模型（Java Memory Model, JMM）定义了Java程序在多线程环境下，共享变量的访问规则，以及线程间如何通信。它是一套抽象规范，屏蔽了不同硬件和操作系统的内存访问差异。
    解决的核心问题：线程间共享变量的可见性、指令重排序
    1. happens-before关系与内存可见性
        1. happens-before的8条规则
        happens-before是可见性的保证，是时间+同步机制，保证执行的结果可以被下一步所见。
            1. 程序顺序规则：同一线程，有数据依赖的多条指令按编码顺序执行，没有数据依赖的可能重排
            2. volatile规则：不同线程中，v变量的写before读
            3. 锁规则：不同线程中，解锁before上锁
            4. 线程start规则：start()before该线程任何动作，是一个线程开始动作的起点
            5. 线程join规则：join()检测其他线程是否结束，检测到了才会开始自己的。
            6. 中断规则：检测中断befor检测到中断才执行的操作。
            7. finalizer规则：构造函数结束before finalizer()。
            8. 传递性：A before B & B before C -> A before C。
        2. 内存可见性
            每个线程拥有私有内存，但对共享变量的修改需要同步到主内存才能让其他线程协作，happens-before关系就通过保持操作的有序性，解决了内存可见性的问题。
        3. 重排序与数据竞争
            编译器、处理器为了优化性能，可能会对指令进行重排序，但如果重排序不合理，多线程环境下可能a线程在b线程修改变量x之前读到不应该读的x，happens-before规则解决了这个问题。
    2. Java内存模型的底层实现
        Java内存模型通过内存屏障（memory barrier）来控制编译器和处理器的重排序行为。内存屏障是一组指令，用于确保特定操作的执行顺序，并强制刷新处理器缓存，保持内存一致性。
        1. 内存屏障
            常见的内存屏障类型是Load、Store之间组合，例如LL、LS、SL和SS，不同的happens-before关系对应不同内存屏障插入方式，比如写v变量前后有SL屏障，先存储，后加载。
        2. 即时编译器优化
            对于不同的处理器，内存屏障的具体实现不同，编译器会根据具体的处理器特性生成机器码，但对JMM规范是没有影响的
        3. 处理器缓存
            内存屏障通过强制刷新缓存，确保共享变量的修改对其他处理器核心可见。例如，当一个线程对共享变量进行修改后，内存屏障会将该变量的值从工作内存同步到主内存，并刷新其他处理器核心中的缓存行，使得其他线程能够读取到最新的值。
    3. 锁与同步
        在单线程中，顺序是获取锁-持有锁时的操作-释放锁-其他操作；多线程中，确保锁释放前的修改对其他线程是可见的（因为其他线程得到锁时，这个锁必然是已经被释放的）
        1. synchronized实现线程同步：锁标志位+Monitor，进入synchronized代码块获取锁，完成这个代码块就是释放锁。
        2. 锁的优化：偏向锁（记录线程id减少同一线程对锁的不停开关的消耗）、轻量级锁（CAS）、重量级锁（Monitor等待队列）。
    4. volatile字段：轻量级线程同步
        1. 对v变量的修改对其他线程立刻可见，编译器、处理器也不会对v变量的读写进行重排序，这时因为JMM会在v变量读写操作前后插入内存屏障，避免了内存可见性问题。
        2. v适用于单个变量的状态标记，比如表示某个任务完成、线程停止等，别的线程可以在执行条件上判断v，从而判断其他线程是否完成工作而自己是是接手或是继续等待。
        3. 不保证复合操作原子性，需要用锁或者其他同步机制。
        4. v读会加载主内存中的最新值，比锁开销更低，但频繁访问主存可能会导致v读造成性能瓶颈。
    5. final字段
        final字段用于构建不可变对象，不可变对象就是被创建之后状态不可修改，变成一个常量。不可变对象因为不会改变状态，所以线程安全。
        1. final字段的写操作before构造函数返回，也就是说，其他线程看到的（已经构造成功）该对象已经是final写过的了。
        2. 具有写操作的有序性（SS防止与构造返回后的操作重排序）、读操作的可见性（读到的时候已经写过了）

### 2. 堆
    对于存储生命周期不固定的数据、或是在多个方法间共享的数据，一般使用堆来存储。
    堆（Heap） 是JVM运行时数据区中最大的一块内存区域，被所有线程共享，用于存放对象实例和数组。几乎所有的对象都在堆中分配内存，也是垃圾回收（GC）管理的主要区域。
    1. 线程共享：堆是公有的，所有线程都能访问到堆中的数据。
    2. 动态分配：运行时申请才知道自己要创建什么对象。
    3. GC管理没有引用的对象。
    4. 物理可以不连续，但逻辑一定连续。
    5. 分代收集理论
        1. 年轻代/年老代：对象创建时，分配在年轻代中，对象年龄过老后，会移动到年老代中。
        2. 新生代会被频繁回收，因为大多数对象很快死亡。少部分长期存活的对象更不容易死，因此会被移入老年代。
        Heap
        │
        ├─ Young Generation
        │    ├─ Eden
        │    ├─ Survivor0
        │    └─ Survivor1
        │
        └─ Old Generation
        对象创建默认在 Eden
        Minor GC 后进入 Survivor
        年龄达到阈值 → Old

### 3. 栈
    栈（Stack） 是JVM运行时数据区中的线程私有内存区域，每个线程创建时都会分配一个栈，用于存储方法调用时的局部变量、操作数栈、动态链接、方法返回地址等信息。栈以栈帧（Stack Frame）为单位，每调用一个方法就压入一个栈帧，方法返回时弹出。
    1. 线程私有：每个线程有自己的栈，互不干扰。
    2. LIFO，先进后出。
    3. 大小固定或者动态
    4. 存储的是方法调用信息，例如局部变量、中间计算结果等，对象引用也在其中。
    5. 栈的方法结束后自己弹出，无须GC。
    Stack Frame
    │
    ├─ Local Variables
    ├─ Operand Stack
    ├─ Dynamic Link
    └─ Return Address

### 4. 方法区
    方法区（Method Area）是JVM运行时数据区中的线程共享内存区域，用于存储已被虚拟机加载的类型信息、常量、静态变量、即时编译器编译后的代码缓存等数据。
    1. 线程共享：方法区是公有的，所有线程都能访问到方法区的数据。
    2，JVM启动时就初始化了方法区，但其物理上不连续，逻辑上连续，因此大小是可以固定也可以扩展的。
    3. JDK7之前用永久代，JDK8后使用元空间，即本地内存，就不在堆里了。

### 5. 垃圾回收GC
    垃圾回收（Garbage Collection, GC） 是JVM自动管理内存的机制，它负责自动检测并回收不再使用的对象，释放内存供新对象使用。程序员无需手动释放内存，避免了C/C++中常见的内存泄漏和野指针问题。
    1. 解决了物理内存泄漏，即没有后续引用的对象肯定会被GC解决。
    2. 没有杜绝逻辑泄露，虽然物理内存会回收，但对象一直被引用。
    3. GC时机不可控，并且GC本身的开销比较大。

### 6. 个人思考
    1. JVM、对象、线程三者的关系
        1. JVM(Java Virtual Machine)是一个在电脑操作系统上运行的进程，向操作系统申请了一部分内存区域（用于存放自己的堆、方法区、栈、PC寄存器等），自己内部还有多个线程。JVM就像一个公司，在土地上有一栋大楼，大楼里有分门别类的资源，供线程们像员工一样使用，完成工作。
        2. 对象是JVM里的数据，对象存放在JVM Heap里。当代码new了一个新的对象，有一个线程执行了该代码，JVM就在堆里分配一点内存给线程来创建该对象的数据，而变量（引用这个对象的指代）在栈中，如果线程执行完毕，栈的对象指代就会回收，C++等语言在这时可能会出现内存泄漏，但JVM会启用GC回收堆中失去指代的对象。
        3. 线程是执行字节码的单位，就像执行代码的员工。程序员写的代码就是甲方给公司的任务，JVM把代码分配给员工线程来执行，所以也会给线程提供一小块内存空间来完成任务，线程有自己的PC寄存器、Java栈、本地方法栈等，但是它们创造出的对象是在共享的堆中，如果谁要调用对象，就要排队等申请，这就涉及了多线程并发问题。
    2. 安全发布
        1. 对象发布（Object Publication）是指使一个对象的引用可以被当前线程之外的其他线程访问到，而不是复制或传递。
        2. JVM为了优化性能，有指令重排序机制，而对象的创建有多个步骤，包括分配内存、执行构造函数、把引用赋给变量，如果与其他线程的指令排序时，其他线程执行顺序插入到执行构造函数之前，那么就有可能访问到没有初始化的对象，这就是不安全发布。
        3. 对象天生是共享的，因为它在谁都可以使用的共享的堆中，但要使用对象，需要叫出它的名字（引用），所以安全发布的本质是安全发布引用。
        4. 对象逃逸：对象的引用被当前方法或线程之外的代码访问到
            1. 一般来说，对象的引用只会存在于自己的作用域，但在返回对象、赋值给成员变量、放进集合、发布给线程等情况中，作用域外的线程有可能访问到。
            2. 但逃逸并不是很严重的事情，不会导致程序不能跑通，但会影响优化。
            3. 发布本身就是逃逸的一种。
